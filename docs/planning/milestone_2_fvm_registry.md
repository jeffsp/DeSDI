## 1. Overview

Milestone 2 establishes the **Immutable Spatial Registry** on the Filecoin Virtual Machine (FVM). It takes the deterministic IPFS CIDs and spatial metadata generated by the Milestone 1 parser and anchors them permanently on-chain. This smart contract layer acts as a decentralized SpatioTemporal Asset Catalog (STAC), enabling researchers to query and locate sharded NASA HDF5 data using geographic bounding boxes and temporal timestamps instead of raw cryptographic hashes.

Additionally, this milestone leverages FVM's unique capabilities to automate data persistence. The contract will proactively monitor Filecoin storage deals and autonomously dispatches renewal payments from a dedicated endowment when data expiration approaches.

## 2. Goals & Non-Goals

### Goals

1. **Develop STAC-compliant smart contracts** — implement Solidity contracts that map spatial bounding boxes `[MinX, MinY, MaxX, MaxY]`, time ranges, and Hilbert cell indexes to IPFS CIDs.
2. **Implement spatial querying logic** — allow on-chain and off-chain (RPC) lookups of datasets based on intersecting geographic coordinates.
3. **Automate storage persistence** — integrate with FVM built-in actors (e.g., Market Actor) to monitor storage deal expiration and autonomously renew deals using contract-held funds (endowment).
4. **Provide deployment tooling** — scaffold a modern EVM development environment (Foundry) configured for Filecoin's Calibration testnet and Mainnet.
5. **Bridge CLI to FVM** — extend the `desdi` CLI from Milestone 1 to parse the generated `manifest.json` and submit registry transactions to the FVM.

### Non-Goals (deferred to later milestones)

- ZKP-backed compute-to-data orchestration (Milestone 3)
- Public web dashboard (Milestone 3)
- Fully automated, large-scale onboarding of petabytes of data (focus is on proving the architecture with sample granules)

## 4. Smart Contract Data Structures

### DataTypes.sol

To efficiently store STAC metadata on-chain, data must be packed to optimize gas costs. 

```solidity
// Represents a STAC Item (Granule)
struct Dataset {
    string id;                  // e.g., "ATL03_20210114000526..."
    int32[4] globalBbox;        // [minLon, minLat, maxLon, maxLat] * 10^6
    uint64[2] timeRange;        // [startEpoch, endEpoch]
    uint64 totalShards;
    address owner;
}

// Represents an IPFS Shard mapping to a Hilbert Cell
struct Shard {
    string cid;                 // CIDv1 string
    uint64 hilbertIndex;        // Hilbert cell ID (from M1)
    int32[4] bbox;              // Cell specific bbox * 10^6
    uint64 dealId;              // Filecoin Storage Deal ID (0 if IPFS only)
}
```

*Note: Floating point coordinates are scaled by `1,000,000` (e.g., `30.26` becomes `30260000`) to be stored as `int32`, which fits precisely and avoids expensive floating-point EVM math.*

## 5. Persistence Management (FVM specific)

The `PersistenceManager.sol` leverages the FVM to interact with Filecoin's Storage Market Actor.

**Workflow:**
1. A public good entity funds the contract's endowment in FIL.
2. Shards are registered and their Filecoin `dealId` is attached.
3. Anyone can call `PersistenceManager.checkAndRenew(uint64 dealId)`.
4. The contract calls the Market Actor to check the deal's expiration epoch.
5. If the deal expires within `< 30 days`, the contract automatically negotiates a new deal (or funds an existing provider) and updates the registry.
6. The caller receives a small FIL bounty to incentivize decentralized monitoring bots (cron replacements).

## 6. CLI Integration

The existing `desdi` CLI from Milestone 1 is extended.

```bash
desdi register output/manifest.json \
  --rpc-url https://api.calibration.node.glif.io/rpc/v1 \
  --private-key $PRIVATE_KEY \
  --contract 0x1234...
```

**Action:** Parses the JSON manifest, scales the bounding boxes to integers, chunks the shards to avoid block gas limits, and submits `registerDataset` and `addShards` transactions to the FVM.

## 8. Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| EVM Gas Limits | Manifests with thousands of shards will exceed block gas limits during registration. | The CLI must chunk shard registration into batches (e.g., 50 shards per transaction) and handle nonces accordingly. |
| Floating Point in EVM | Solidity does not support floats, breaking BBox storage. | Scale coordinates by 10^6 and store as `int32`. |
| FVM API Upgrades | Filecoin's built-in actor APIs can shift between network versions. | Abstract Market API interactions behind interfaces. Use well-supported FVM wrapper libraries if available. |
